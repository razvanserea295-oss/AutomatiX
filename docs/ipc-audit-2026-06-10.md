# Audit IPC — „no handler registered for …" — 2026-06-10

**Scop:** cross-check sistemic al *tuturor* apelurilor IPC din frontend (`src/**`) împotriva registry-ului de comenzi backend, pentru a localiza cauza erorilor de tip **`No handler registered for '…'`**.

**Metodă:** scanare statică deterministă (script `scripts/ipc-audit.cjs`, read-only). Extrage numele de comenzi din:
- **Apeluri FE:** `apiCommand<…>('cmd')`, `electron.invoke('cmd')`, `fetch('/api/cmd/cmd')`.
- **Înregistrări BE:** `ipcRegister<…>('cmd', …)` în `electron/**` + `server/**`, plus `ipcMain.handle('cmd')` (Electron-only).
- Verifică în plus dacă funcția `registerXxxHandlers()` care conține un `ipcRegister` este efectiv *apelată* în `registerAllHandlers()` (`electron/ipc/handler.ts`).

Rezultatul brut JSON: `scripts/ipc-audit-result.json`.

> ⚠️ Audit static la momentul `2026-06-10`. Două sesiuni Claude editează în paralel același working tree — datele reflectă fișierele **salvate pe disc**, nu buffere needitate din editor.

---

## 0. Corecție de premisă: există UN SINGUR registry, nu două

Misiunea presupunea „ambele registry-uri". În realitate (vezi `CLAUDE.md` + cod):

- `electron/commands/registry.ts` ține **o singură** hartă `Map<string, handler>` populată de `ipcRegister(name, handler)`.
- `server/commandRouter.ts` **NU** are un registry paralel — citește *aceeași* hartă prin `getCommand(name)`. Eroarea `No handler registered for '…'` se aruncă la [`server/commandRouter.ts:37`](../server/commandRouter.ts#L37) când `getCommand(name)` întoarce `undefined`.
- Singura „dublură" este modul de instalare: `ipcRegister` populează harta **și** (doar în Electron real) cheamă `ipcMain.handle`. Cum Electron a fost scos, partea `ipcMain` e inertă; contează doar harta.

**Consecință:** o comandă e disponibilă peste HTTP dacă și numai dacă `ipcRegister('cmd', …)` a fost *executat* la boot. Nu există divergență „registry A vs registry B".

## 2. ❌ Comenzi apelate dar neînregistrate (cauza directă a erorilor)

**ZERO.** Toate cele 287 de comenzi dispatch-uite prin HTTP (`apiCommand` + `fetch /api/cmd/*`) au un `ipcRegister` corespunzător, case-sensitive, în registry.

Comenzile care păreau „lipsă" la o primă scanare (`create_project`, `create_client`, `update_invoice_status`, …) **sunt** înregistrate — folosesc generice imbricate `ipcRegister<Cmd<CreateProjectRequest>>('create_project', …)`; un regex naiv `<[^>]*>` le ratează. Sunt confirmate înregistrate și prin faptul că `wrapCommand`/`withSse` din `commandRouter.ts` le împachetează — iar `wrapCommand` **aruncă** dacă numele nu e în registry (deci boot-ul ar fi căzut altfel).

## 5. 🔴 Cauza reală a erorilor „no handler registered" (runtime)

Întrucât §2 arată zero lipsuri statice, erorile sunt de **runtime**. În ordinea verosimilității:

### 5.1 Server nereparnit după edit backend (cauza #1 cu 2 sesiuni paralele) — FOARTE PROBABIL
Aplicația web servește bundle-ul **construit** (`npx vite build` → `dist/`). Fluxul tipic:
1. O sesiune adaugă în FE un `apiCommand('get_foo')` **și** `ipcRegister('get_foo', …)` în backend.
2. Frontend-ul e re-buildat (sau HMR) → browserul cheamă `get_foo`.
3. Dar **procesul server Node nu a fost repornit** → registry-ul din memorie încă nu conține `get_foo` → `No handler registered for 'get_foo'`.

Simetric: un FE rebuildat care cheamă o comandă **redenumită** lovind un server vechi. Cu două sesiuni care editează + rebuildează necoordonat, e scenariul cel mai plauzibil. `dist/` (18:07) e mai nou decât `src/` (17:44) — semn de rebuild-uri frecvente.

**→ Remediu:** după *orice* edit în `electron/**` sau `server/**`, **repornește `npm run server`**. Rebuild-ul de FE singur nu reîncarcă handler-ele backend.

### 5.2 Agregator „totul-sau-nimic" — înregistrare cascadată ratată — PROBABIL ✅ REPARAT (2026-06-10)
[`server/commandRouter.ts:82-88`](../server/commandRouter.ts#L82) învelește **întregul** `registerAllHandlers()` într-un singur `try/catch`, iar [`electron/ipc/handler.ts:73-145`](../electron/ipc/handler.ts#L73) cheamă cele ~60 de `registerXxxHandlers()` **secvențial, fără try/catch individual**.

**Consecință:** dacă UN singur modul aruncă la înregistrare (import rupt, eroare la module-scope, o sesiune a lăsat fișierul într-o stare invalidă), **toate** modulele de DUPĂ el în secvență nu se mai înregistrează, iar singurul semn e un warning: `[server] IPC aggregator failed; some commands may be unregistered`. Rezultă un val de `No handler registered for '…'` — exact tiparul „mai multe comenzi `get…` deodată".

Ex.: dacă `registerSalesHandlers()` aruncă, dispar din registry toate de la sales încolo (deplasari, checklist, ai, search, maintenance, handoffs, pdf, quotations, calendar, timeTracking, serviceTickets, portal, threeWayMatch, rfq, notifications, demoSeed, anaf, signatures, reports, personalTasks, goodsReceipts, userSessions, broadcasts, activityLog).

**→ REPARAT:** `electron/ipc/handler.ts` — `registerAllHandlers()` rulează acum fiecare registrar într-un `try/catch` propriu (array `REGISTRARS` `[label, fn]`). Un modul defect își pierde **doar** comenzile lui și este logat explicit (`[ipc] registrar 'X' FAILED …` + sumar la final), restul modulelor se înregistrează normal. Verificat: `tsc -p tsconfig.server.json` = 0 erori; rulare reală → `Handler registration complete`, 398 comenzi, 57/57 registrar-e OK.

### 5.3 Race de editare concurentă — TRANZITORIU
Două sesiuni care scriu simultan în `handler.ts` / fișiere `ipc/*` → stare temporar invalidă (un import pe jumătate scris) la momentul boot-ului serverului. Se rezolvă de la sine la următorul boot cu sursă coerentă.

### 5.4 Buffere needitate (nesalvate)
Eroarea poate proveni din cod **nesalvat** în editorul celeilalte sesiuni (build din buffer ≠ disc). Audit-ul vede doar discul; nu pot infirma asta static.

## 6. Recomandări

### 6.1 Imediat (oprește erorile acum)
- **Repornește serverul** (`npm run server`) după edituri backend. Verifică la boot linia `[server] N commands registered` — dacă `N` scade brusc între reporniri, un modul a căzut la înregistrare (§5.2).
- Caută în logul serverului warning-ul `IPC aggregator failed` — dacă apare, acolo e modulul rupt.

### 6.2 Hardening `registerAllHandlers()` — ✅ APLICAT (2026-06-10)
Apelurile secvențiale au fost înlocuite cu un loop cu try/catch per-modul (array `REGISTRARS: [label, fn]`) în `electron/ipc/handler.ts`. Un modul defect pierde doar comenzile lui (logat explicit), nu și pe cele înregistrate ulterior. Vezi §5.2.

### 6.3 Gardă de boot (diagnostic permanent)
La pornirea serverului, după înregistrare, loghează diferența *comenzi-FE-așteptate vs registry*. Se poate genera o listă albă din acest audit. Cel mai simplu: păstrează `scripts/ipc-audit.cjs` și rulează-l în CI / pre-commit — pică build-ul dacă apare o comandă `apiCommand` fără `ipcRegister`.

### 6.4 Curățenie (low priority)
- §3: decide pentru cele 20 desktop-only — fie le expui prin `ipcRegister`+`apiCommand` (dacă vrei audit/server-control/updater în web), fie le marchezi clar ca Electron-legacy.
- §4: triază cele 117 comenzi moarte per-feature; leagă UI-ul lipsă (RFQ, three-way-match, engineering tree, goods receipts) sau marchează-le ca backend-ready-pending-UI.
